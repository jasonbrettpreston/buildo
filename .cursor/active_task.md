## Technical Implementation

### Part 1 — DOWN Block Classification

After reading each of the 40 migrations, each gets one of three DOWN block types:

| Type | When | DOWN Comment |
|------|------|-------------|
| **Table drop** | `CREATE TABLE` | `-- DROP TABLE IF EXISTS {table};` + any indexes |
| **Column drop** | `ALTER TABLE ADD COLUMN` | `-- ALTER TABLE {tbl} DROP COLUMN IF EXISTS {col};` |
| **Forward-only** | Data migrations (`UPDATE`, `INSERT`, `DELETE`), materialized views needing manual rebuild, or where reversal would require restored data | Explicit comment: `-- Data migration — forward-only. Restore from backup if rollback required.` |

Migrations already confirmed to have DOWN blocks (excluded from work): **030**, **033**, **037**, **039**.
Migration 034 (materialized view) gets: `-- DROP MATERIALIZED VIEW IF EXISTS mv_monthly_permit_stats;`
Migrations 028 (INSERT INTO trades), 029 (UPDATE trades data): forward-only with explanation.

### Part 2 — Schema Spec Refresh

`npm run db:docs` queries `information_schema` read-only and replaces the `<!-- DB_SCHEMA_START -->...<!-- DB_SCHEMA_END -->` block in `docs/specs/00-architecture/01_database_schema.md` with the current 47-table schema. Requires live DB — run after confirming `pg_isready`.
