## Phase 5 — Pattern Routing

Two recurring shapes surfaced in this audit, both routable to stronger destinations:

### Pattern A: Spec mandate exists but no enforcement test (HIGH H2 + HIGH H3 both)
**Shape:** Spec §X says "every Y MUST do Z" but neither a regression test nor a CI lint rule grep-asserts the presence of Z. Two examples this audit:
- §7.3 mandates `track('route_decision', ...)` from AuthGate / AppLayout — no test grep-asserts the calls exist
- §8.3 mandates `isFetching` toggle tests — no test exists for any render gate

**Routing:** Add §9.21 (NEW) — a `mobile/__tests__/spec99.mandates.lint.test.ts` that statically asserts every `## §X mandate` in Spec 99 has a corresponding implementation grep-target. Pattern matches the existing `routerHygiene.lint.test.ts` and `storeReset.coverage.test.ts` static-shape style.

### Pattern B: Spec text drifts when code is hardened beyond spec (M1 + M2 both)
**Shape:** A WF3 fix tightens code beyond the spec text (e.g., spec says `removeQueries`, code uses `clear()`; spec says `user_id !== uid`, code adds `!user_id` falsy guard). The spec text becomes stale documentation.

**Routing:** Add §9.22 (NEW) — quarterly Spec 99 sync WF that diffs the §3-§7 rule text against the actual implementation paths. Add a `## Hardening Beyond Spec` subsection to §B5 + §5.2 explicitly listing intentional over-specification.

Single-shot findings (H4 mobile typecheck + H5 MEMORY.md staleness) file as individual WF3s, not pattern-routed.
